# Dependency Marginal Ledger

Bead: `asupersync-dep-p1-foundations-upksjk.2`

## Purpose

The canonical machine-readable ledger is
`artifacts/dependency_marginal_ledger_v1.json`. It is generated by the
Cargo-built Rust binary at `src/bin/dependency_marginal_ledger.rs` and checked
by `tests/dependency_marginal_ledger_contract.rs`.

The ledger measures the external package-version set attributable to each
active direct dependency edge of the root `asupersync` package. It does not
estimate a marginal by deleting one node from an already-resolved graph.
Instead, the generator writes a second valid Cargo manifest with exactly that
root edge and its feature references removed, seeds it from the independently
resolved baseline lockfile, and asks Cargo to resolve the counterfactual
offline.

## Canonical Matrix

The canonical feature profiles are:

- `minimal`: no default features.
- `default`: the normal default feature vector.
- `tls`, `sqlite`, `kafka`, `metrics`, `cli`, `compression`,
  `trace-compression`, `io-uring`, and `loom-tests`: defaults plus the named
  feature.
- `fuzz-quarantine`: defaults plus `fuzz`; this is explicitly not a production
  graph.
- `workspace-dev-build-audit`: a shadow copy of every workspace member
  manifest, all features enabled, and dev/build edges retained.

Every profile is resolved for:

- `x86_64-unknown-linux-gnu`
- `aarch64-apple-darwin`
- `x86_64-pc-windows-msvc`
- `wasm32-unknown-unknown`

Normal profiles use a synthesized out-of-workspace consumer. The workspace
audit copies only Cargo manifests and creates inert dummy targets in the owned
work directory. Cargo therefore sees all workspace members and their
dependency declarations without compiling or executing repository source.
External reachability is unioned from every workspace member. The
counterfactual still removes only the selected direct edge from the root
package, so a package retained by another member is correctly not counted as a
workspace-wide marginal.

## Stable Keys and Counts

Each marginal row is keyed by the exact tuple:

`feature_profile × target_triple × host_triple × dependency_edge_kind × direct_root_edge`

`direct_root_edge` distinguishes normal, dev, build, and target-qualified
manifest entries, including renamed dependencies. The ledger carries both
package-version counts and unique package-name counts. Package IDs from
Cargo's resolve graph define reachability and set difference; package-name
text never substitutes for a package ID.

The repository `Cargo.lock` seeds every baseline. The resolved baseline lock
then seeds each exact counterfactual. This holds the upstream version choice
constant while still making Cargo resolve the edited manifest. Each row
records baseline and counterfactual manifest hashes, lockfile hashes, package
counts, marginal package IDs, and the exact Cargo commands.

First-party workspace path packages and the two synthetic roots are excluded.
An unrelated external path package is not silently merged into that
first-party set.

## Host and Target Context

The requested target triple controls Cargo's target graph. Build dependency
edges and proc-macro roots are labeled `host`; normal dependencies are labeled
`target`; dev edges are labeled `target-dev`. The host triple comes from
`rustc -vV` and is part of every result key.

This is a resolution ledger. A host-context label does not claim that the build
script or proc macro was compiled or executed.

## Upstream Identity

Marginal package versions carry normalized upstream identities:

- repository URLs normalize scheme and host case, trailing slash, and `.git`;
- registry fallbacks retain the registry source and checksum, with a literal
  `missing-checksum` state;
- Git sources retain the normalized URL and immutable revision;
- a package without usable source metadata falls back to
  `unknown:<full-package-id>`.

Unknown packages therefore fail closed per package ID. They are never grouped
under a shared generic `unknown` identity.

## Native-Code Evidence

`root_native_code` describes known native behavior anywhere under the selected
direct root closure. `marginal_native_code` describes only the packages removed
by that counterfactual. Their statuses are:

- `active`: a package-specific rule identifies a resolved native build or link
  boundary, such as `rdkafka-sys` configure/make or bundled
  `libsqlite3-sys`.
- `declared-inactive`: a known native declaration exists but its activating
  edge is absent. The `signal-hook` without `cc` fixture guards this state.
- `unknown`: Cargo exposed an unclassified custom-build target.
- `none`: no package in the inspected set produced native evidence.

Cargo metadata alone cannot prove that native code compiled. These statuses are
attribution receipts, not safety certificates.

## Taxonomy and Phase Forecasts

When a direct dependency is an incumbent in
`artifacts/dependency_safety_taxonomy_v1.json`, the row embeds the stable
candidate ID, safety class, sensitivity tags, program phase, and program
verdict. Unmapped dependencies use
`unsafe_exposure_class = "unclassified-fail-closed"`.

Phase forecasts are generated from the measured individual-edge rows. Their
lower bound is the union of individual marginal package IDs; their upper bound
is the sum of individual marginal counts. The interval is intentionally not an
exact simultaneous-removal forecast. Only a separately resolved multi-edge
counterfactual could support that claim.

## Regeneration

The artifact's `source_commit` identifies the commit containing the generator,
contract, documentation, taxonomy, root manifest, and lockfile used to produce
it. From a clean `main`, regenerate with remote Cargo execution:

```bash
GENERATOR_COMMIT="$(git rev-parse HEAD)"
RCH_WORKER=vmi1227854 RCH_REQUIRE_REMOTE=1 rch exec -q \
  --base "$GENERATOR_COMMIT" --clean-overlay --no-overlay -- \
  env CARGO_TARGET_DIR="${RCH_TARGET_BASE:-${TMPDIR:-/tmp}}/rch_target_dependency_marginal_ledger_generate" \
  CARGO_INCREMENTAL=0 CARGO_PROFILE_DEV_DEBUG=0 \
  RUSTFLAGS='-D warnings -C debuginfo=0' \
  cargo run -j 8 --quiet -p asupersync \
  --bin dependency_marginal_ledger \
  --no-default-features --features dependency-ledger -- \
  --source-commit "$GENERATOR_COMMIT" --jobs 8 --output - \
  > artifacts/dependency_marginal_ledger_v1.json
```

The redirect captures the generator's stdout locally; RCH status and
diagnostics remain on stderr. `--jobs` changes only bounded counterfactual
concurrency. Rows are sorted by their stable key before serialization.

Validate the generated artifact with:

```bash
jq empty artifacts/dependency_marginal_ledger_v1.json
RCH_REQUIRE_REMOTE=1 rch exec -- \
  env CARGO_TARGET_DIR="${RCH_TARGET_BASE:-${TMPDIR:-/tmp}}/rch_target_dependency_marginal_ledger_contract" \
  CARGO_INCREMENTAL=0 CARGO_PROFILE_TEST_DEBUG=0 \
  RUSTFLAGS='-D warnings -C debuginfo=0' \
  cargo test -p asupersync --test dependency_marginal_ledger_contract -- --nocapture
```

## No-Claim Boundaries

Passing the contract establishes the schema, canonical matrix, key uniqueness,
Cargo-command shape, count arithmetic, counterfactual provenance, taxonomy
links, native-evidence states, phase-forecast arithmetic, and documentation
markers.

It does not prove compilation for any target, runtime correctness,
interoperability, exploitability, performance improvement, release readiness,
broad workspace health, live RCH fleet availability, or permission to replace
a dependency. The fuzz profile is quarantined evidence, not a production
consumer guarantee. Individual-edge marginals are not additive savings claims.
